# Jump Space 危机系统深度分析

## 一、损坏(Broken)与致命损坏(Critical)差异

### 1.1 三态系统

```
  Healthy (0)                    Broken (1)                    Critical (2)
  ┌─────────┐   耐久 < BrokenThreshold   ┌─────────┐   CrisisHandler选中   ┌──────────┐
  │ 正常工作  │ ──────────────────────→ │ 故障停工  │ ─────────────────→ │ 临界崩溃  │
  └─────────┘                          └─────────┘                      └──────────┘
       ↑                                    │                               │
       │         玩家修复(RepairTool)         │       玩家稳定(Stabilize)        │
       └────────────────────────────────────┘───────────────────────────────┘
                                            ↑                               │
                                            └─── 稳定成功 ←─── ITimingInteraction倒计时
                                                                            │
                                                                    超时 → 永久摧毁
```

### 1.2 五维度对比

| 维度 | Broken (1) | Critical (2) |
|------|-----------|-------------|
| **功能影响** | 组件停工，不再执行功能 | 组件停工，不再执行功能 |
| **触发方式** | 耐久度 < `BrokenThreshold × MaxDurability`（默认30%） | `CrisisHandler.PickComponentToDamage()` 选中，调用 `SetCriticalServerRpc(2)` |
| **玩家交互** | Repair — 无计时，花费 `MateriaCostToRepair` | Stabilize — 有ITimingInteraction倒计时，花费 `MateriaCostToStabilize` |
| **后果** | 安全状态，可随时修复 | 超时未稳定 → 组件**永久摧毁** |
| **视觉效果** | `BrokenComponentVisuals` 基础损坏效果 | 增强损坏效果 + `EngineBurnEffectComponent`（引擎燃烧特效） |

### 1.3 关键代码路径

- **Broken判定**: `IsComponentBrokenDisabledOrBlockedForUse` → `m_ComponentHealthStatus != Healthy(0)` → return true
- **Critical触发**: `CrisisHandler.PickComponentToDamage()` → `SetCriticalServerRpc(1)`(Broken) 或 `SetCriticalServerRpc(2)`(Critical)
  - `m_OnlyDamageComponentsNeverSetCritical == true` → 只设Broken(1)
  - `m_OnlyDamageComponentsNeverSetCritical == false` → 设Critical(2)
- **Critical交互**: `CriticalComponentInteraction.OnHealthStatusChanged(Critical)` → `EnableInteraction` → ITimingInteraction倒计时小游戏
  - 成功(Success) → 组件回到Broken状态
  - 失败 → 组件永久摧毁

## 二、节流危机机制

### 2.1 机制概述

`CrisisHandler` 在每次 `Update()` 中评估是否触发危机(`TriggerCrisis()`)，但当 AI Director 的**压力水平**超过阈值时，会**跳过**危机触发——这就是"节流危机"。

### 2.2 节流判定

```
if (GetNormalizedStressLevel() >= m_ThrottleCrisisIfStressIsHigherThan)
    → 跳过 TriggerCrisis()  // 节流：压力太高，不再新增危机
else
    → 正常评估 TriggerCrisis()
```

- `m_ThrottleCrisisIfStressIsHigherThan` 默认值 = **1.0**（即不节流，因为归一化压力永远 ≤ 1.0）
- 配置为 0.70 时：压力 ≥ 70% 就停止新增危机

### 2.3 压力曲线算法（StressCurveHandler）

#### 5个压力分量直接求和

| 分量 | 计算方式 | ISIL常量 |
|------|---------|----------|
| **SpaceCombatStress** | 遍历 `AI_Controller_NearSpace_Base`，战斗状态敌舰各 + 常量 | `[18C899A64]` |
| **OnFootCombatStress** | 遍历 `OnFootTrackerBlackboardData`，战斗状态步行AI各 + 常量 | ISIL内联 |
| **DownedStress** | 遍历玩家列表 `IsDowned`，返回倒地玩家数 (int→float) | 直接计数 |
| **ShipHealthStress** | 基于 `ShipHealth01` 的阶梯函数（5档） | ISIL内联 |
| **CriticallyDamagedDamagepointsCount** | 临界损坏点计数 (int→float) | 直接计数 |

#### ShipHealthStress 阶梯函数

```
ShipHealth01 (0~1) → 5档固定应力值：

  血量区间          应力值
  ≥ 0.8            → 0.0
  [0.6, 0.8)       → 1.0
  [0.4, 0.6)       → 2.0
  [0.2, 0.4)       → 3.0
  < 0.2            → 4.0
```

#### 平滑插值

```csharp
// UpdateStressLevel() 每秒调用一次
float current = CurrentStressLevel;           // 5分量直接求和
float interpolated = m_InterpolatedStressLevel;

if (current >= interpolated)
    interpolated = current;                    // 上升：即时响应
else
    interpolated += (current - interpolated) * lerpFactor;  // 下降：缓慢衰减

m_InterpolatedStressLevel = interpolated;
```

#### 归一化

```csharp
NormalizedStressLevel = Clamp(InterpolatedStressLevel * scaleFactor, 0, 1.0);
```

- `scaleFactor` 由配置决定，将原始应力值映射到 [0, 1]
- 最终 `GetNormalizedStressLevel()` 返回 `DirectorBlackboardData.m_NormalizedStressLevel.Value`

## 三、伤害点（DamagePoint）子系统

### 3.1 定义

**伤害点**（`DamagePoint_Base`）是飞船内部的**可损坏位置点**，与**组件**（`SpaceShip_BaseComponent`，如引擎/发电机）是**两套独立系统**。

- 每个DamagePoint有自己的 `HealthComponent_Base`，独立追踪血量
- 常量 `CriticallyDamagedThreshold = 0.8` — **血量损失达80%时进入临界损坏状态**
- `IsCriticallyDamaged` 属性从 `m_DamagePointBlackboardData.m_IsCriticallyDamaged`（ReactiveProperty）读取
- Awake中订阅血量变化：`health >= CriticallyDamagedThreshold(0.8)` → 标记为临界

### 3.2 6种伤害点子类

| 子类 | 说明 | 特殊机制 |
|------|------|---------|
| `DamagePoint_FuelLine` | 燃料管 | 分段损坏/修复 |
| `DamagePoint_FuelLineDestruction` | 燃料管破坏 | 不可修复的终极损坏 |
| `DamagePoint_Radiator` | 散热器 | 实现 `IFireNode`，可起火/灭火 |
| `Damagepoint_MateriaLeak` | 物质泄漏 | 泄漏效果 |
| `Damagepoint_RadiationLeak` | 辐射泄漏 | 辐射区域效果 |
| `Damagepoint_Shipcore` | 船核 | 最关键的损坏点 |

### 3.3 与CrisisHandler的关系

CrisisHandler在 `PickComponentToDamage()` 中根据飞船血量选择目标：

```
ShipHealth01 > m_HealthPercentUntilDamageDamagePoints
    → 优先选 DamagePoint（伤害点）

ShipHealth01 ≤ m_HealthPercentUntilDamageDamagePoints
    → 转而选 Component（引擎/发电机等）
```

- `CriticallyDamagedDamagepointsCount` 属性统计当前临界损坏点数量
- 该计数作为**压力曲线第5分量**，影响AI Director节流判定
- 临界损坏点越多 → 压力越高 → 越可能触发节流（停止新增危机）

### 3.4 DamagePoint vs Component 对比

| 维度 | DamagePoint（伤害点） | Component（组件） |
|------|---------------------|------------------|
| 基类 | `DamagePoint_Base` → `NetworkedObject_Base` | `SpaceShip_BaseComponent` → `NetworkedObject_Base` |
| 血量系统 | `HealthComponent_Base`（独立血量） | `ComponentDurabilityBlackboardData`（耐久度） |
| 临界阈值 | `CriticallyDamagedThreshold = 0.8`（80%损失） | `BrokenThreshold = 0.3`（30%损失→Broken） |
| 临界触发 | 血量损失 ≥ 80% 自动标记 | CrisisHandler 选中 → `SetCriticalServerRpc(2)` |
| 修复方式 | `Repair(float points)` 恢复血量 | Repair(Broken) / Stabilize(Critical) |
| 可被CrisisHandler选中 | `m_CanBeRandomlyPickedForCrisishandler` | 非Invulnerable + 非已Critical |
| 影响压力曲线 | 临界计数 → 压力第5分量 | 不直接计入压力 |
