# Jump Space 组件故障（Malfunction）机制完整分析

## 一、架构总览：双层系统

组件故障系统分为**静态配置层**和**运行时状态层**，两者分离设计：

| 层级 | 数据来源 | 关键字段 |
|------|---------|---------|
| 静态配置 | `Spaceship_CompData_Base` | `BrokenThreshold`(0.3)、`m_CostToRepair`、`m_IsInvulnerable`、`MateriaCostToStabilize` |
| 运行时状态 | `ComponentDurabilityBlackboardData` | `m_Durability`、`m_MaxDurability`、`m_Durability01` |
| 运行时状态 | `BaseComponent_Blackboarddata` | `m_ComponentHealthStatus`、`m_ComponentPowerState`、`m_IsActive` |

## 二、完整故障触发链

```
1. 船体受伤害
   └─ Playership_DamageController.OnShipTakenDamage(DamageInfo)
      └─ 记录伤害时间戳 + 累积伤害值

2. CrisisHandler.Update() 每帧评估
   └─ EvaluateAndTriggerCrisis()
      ├─ 检查: 累积伤害 ≥ m_AccumulatedDamageThresholdUntilTriggerCrisis ?
      ├─ 检查: 距上次危机 ≥ m_MinTimeBetweenCrises ?
      └─ 检查: 当前危机数 < m_MaxAmountOfSimultaneousCrises ?

3. 触发危机 → TriggerCrisis()
   ├─ CanTriggerComponentFailure() → 选组件故障
   │   ├─ 船体血量 < m_HealthPercentUntilDamageComponents ?
   │   ├─ 距上次组件Critical ≥ m_MinTimeBetweenComponentFailures ?
   │   ├─ 筛选: 非Invulnerable + 非已Critical的组件
   │   ├─ 优先: 血量 < m_HealthThresholdToPickEnginesAndGenerator → 选引擎/发电机
   │   └─ 一次最多 m_MaxAmountOfComponentsGoingCriticalInOneGo 个
   │
   ├─ CanTriggerDamagePoint() → 选损坏点
   │   ├─ 船体血量 < m_HealthPercentUntilDamageDamagePoints ?
   │   ├─ 距上次损坏点 ≥ m_MinTimeBetweenDamagepointFailure ?
   │   └─ 筛选合法损坏点(FuelLine/Radiator/Repair/Shipcore)
   │
   └─ CanStartFire() → 起火
       ├─ 船体血量 < m_HealthPercentUntilStartFires ?
       ├─ 当前火灾数 < m_MaxAmountOfFires ?
       └─ 距上次起火 ≥ m_TimeUntilFireSpread ?
```

## 三、组件健康三态流转

```
  Healthy (0)                    Broken (1)                    Critical (2)
  ┌─────────┐   耐久 < BrokenThreshold   ┌─────────┐   CrisisHandler选中   ┌──────────┐
  │ 正常工作  │ ──────────────────────→ │ 故障停工  │ ─────────────────→ │ 临界崩溃  │
  └─────────┘                          └─────────┘                      └──────────┘
       ↑                                    │                               │
       │         玩家修复(RepairTool)         │       玩家稳定(Stabilize)        │
       └────────────────────────────────────┘───────────────────────────────┘
```

- **Healthy → Broken**: 组件耐久度(`m_Durability`)低于 `BrokenThreshold × m_MaxDurability`（默认30%）
- **Broken → Critical**: `CrisisHandler.PickComponentToDamage()` 选中该组件，调用 `SetComponentHealthStatus(Critical)`
- **Critical → Broken**: 玩家使用修复工具稳定（Stabilize），花费 `MateriaCostToStabilize`
- **Broken → Healthy**: 玩家修复（Repair），花费 `MateriaCostToRepair`

## 四、损坏点（DamagePoint）子系统

损坏点有独立生命值，是船体内部物理系统：

| 损坏点类型 | 功能 | 特殊机制 |
|-----------|------|---------|
| `DamagePoint_FuelLine` | 燃油管路 | 分段损坏/修复，加压/泄压 |
| `DamagePoint_Radiator` | 散热器 | 可起火/灭火，实现IFireNode |
| `DamagePoint_Repair` | 修理站 | 关联RepairTool_Collider |
| `Damagepoint_Shipcore` | 船体核心 | 最关键的损坏点 |

损坏点生命值 < `CriticallyDamagedThreshold`(0.8) → Critical状态

## 五、内部损坏类型（InteriorDamageType）

14种内部损坏事件，由危机系统触发：

`Gasleak` / `Fire` / `Hullbreach` / `Fuelleak` / `PowerLeak` / `ComponentFailure` / `RadiatorFailure` / `Radiation` / `Emp` / `CoolantLeak` / `OxygenLeak` / `ToxicLeak` / `ElectricalFailure` / `StructuralDamage`

## 六、玩家交互与修复

- **VAssist组件舱** (`VAssist_ComponentPod`): 显示组件状态，提供修复/稳定/卸载/替换操作
  - Broken组件 → 显示"Repair"选项（花费 `MateriaCostToRepair`）
  - Critical组件 → 显示"Stabilize"选项（花费 `MateriaCostToStabilize`）
- **CriticalComponentInteraction**: Critical组件的定时小游戏交互
- **神器加成**: `Artifact_ComponentStabilizer` 自动稳定Critical组件，`Artifact_TelmarianStabilizer` 降低稳定费用

---

## 七、视觉反馈

- `BrokenComponentVisuals`: Broken/Critical组件显示火花、灯光闪烁、损坏材质
- `GUI_Malfunctionbox`: HUD故障面板，显示引擎临界/损坏/无电力/燃油管路故障等警告
- `EngineBurnEffectComponent`: 引擎Critical时燃烧特效，稳定后消除

## 八、关键配置参数（DamageControllerData）

| 参数 | 含义 |
|------|------|
| `m_MaxAmountOfSimultaneousCrises` | 最大同时危机数(1~5) |
| `m_AccumulatedDamageThresholdUntilTriggerCrisis` | 触发危机累积伤害阈值 |
| `m_MinTimeBetweenCrises` | 危机最小间隔(秒) |
| `m_MinTimeBetweenComponentFailures` | 组件故障最小间隔(秒) |
| `m_MinTimeBetweenDamagepointFailure` | 损坏点失效最小间隔(秒) |
| `m_OnlyDamageComponentsNeverSetCritical` | 只损坏不设Critical |
| `m_MaxAmountOfComponentsGoingCriticalInOneGo` | 一次最多几个Critical |
| `m_HealthPercentUntilDamageComponents` | 开始损坏组件的血量% |
| `m_HealthPercentUntilDamageDamagePoints` | 开始损坏损坏点的血量% |
| `m_HealthThresholdToPickEnginesAndGenerator` | 优先选引擎/发电机的血量阈值 |

---

**结论**: 组件故障是**纯运行时动态系统**。CompData只存储静态配置（阈值/费用/免疫），实际耐久度和健康状态在运行时通过Blackboard数据驱动。ModuleExporter从CompData静态导出只能获取 `BrokenThreshold`、`m_CostToRepair`、`m_IsInvulnerable` 等配置参数，无法导出运行时耐久值。
